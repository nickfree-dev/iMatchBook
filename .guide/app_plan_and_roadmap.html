🎯 Vision & Goal

What iMatchBook Needs to Become

iMatchBook is evolving from a single-purpose receipt matcher into a unified real estate rental bookkeeping platform. Inspired by the best features of Stessa, Landlord Studio, and Baselane, the goal is:

  • 📥 Import bank statements (CSV, QIF, OFX) and categorize transactions
  • 📸 Capture & link receipts and invoices (camera or file upload) with AI-powered OCR
  • 🏘️ Track per-property income and expenses across a rental portfolio
  • 📊 Generate reports: P&L, Schedule E, cash flow by property and date range
  • 🔒 Secure multi-user access with JWT authentication
  • 📱 Work on any device as an installable PWA with offline capability

Full Stack Decision Matrix

Backend PHP 8.1 (Custom slim API) Already established. Avoid adding Laravel to keep it lightweight. Just refactor into proper Model/Controller/Util classes.

🔐 Authentication Plan

JWT with HttpOnly Refresh Token

sequenceDiagram participant FE as React App participant BE as PHP API participant DB as MySQL FE->>BE: POST /api/auth/login {email, password} BE->>DB: Verify user credentials (bcrypt_verify) DB-->>BE: User record BE-->>FE: Access Token (JWT, 15min) + Refresh Token (HttpOnly cookie, 7d) FE->>BE: GET /api/transactions (Authorization: Bearer accessToken) BE-->>FE: 200 OK + data Note over FE,BE: Access Token expires... FE->>BE: POST /api/auth/refresh (sends cookie automatically) BE-->>FE: New Access Token

Frontend Auth Flow

  • Access token stored in React state / memory only (never localStorage to prevent XSS)
  • Axios interceptor attaches token to every request header
  • On 401 response, Axios automatically calls /api/auth/refresh and retries
  • <ProtectedRoute> component wraps all authenticated pages
  • React Context provides useAuth() hook: { user, login, logout, isAuthenticated }

Backend Auth Implementation

  • Use firebase/php-jwt (add to Composer)
  • middleware/auth.php: validates Bearer token on every protected endpoint
  • Refresh token stored in DB (allows forced logout by deleting DB record)
  • Passwords hashed with password_hash($pass, PASSWORD_BCRYPT)

  • Extracts: merchant, date, total, payment type, card last 4, line items
  • Proximity matcher (±3 days, exact amount)
