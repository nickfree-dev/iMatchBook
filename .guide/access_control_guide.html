3. Centralized Role Enforcement

✅ Unified Security: Role enforcement is now centralized in router_security.php. Pages no longer need their own if (!is_admin()) or if (!$is_instructor) guards.

The routing flow follows this logic:

  1. Lookup: findPageRole($name) returns the context and required role.
  2. Authenticate: Calls authorize() via the Auth2 system.
  3. Verify Role: Router checks $securityLevel or identity flags against config.
  4. Deny: If role is insufficient, router calls show_lostPageMessage() with a 403 status.

6. Question Router Flow

This router is more complex - it handles question rendering with multiple access layers:

  1. User Login: checkUserLoginAndSetUserInfo(true) for non-preview requests
  2. Course Access: getUserCourseAccessDetails($COURSE) → must return securityLevel >= 0
  3. Assignment Access: checkUserAccessToAssignment() for students
  4. Prerequisite Check: Questions hidden until prereqs are satisfied
  5. Password Protection: For quizzes with passwords
  6. Time Limits: For timed quizzes

8. Key Authentication Functions (Auth2 System)

The application has migrated to the Auth2 system for centralized identity and course access management.

FunctionLocationPurpose
authorize($policy, $course) utilities_auth2.php Resolves identity/course access using a specific policy (e.g., require_login).
handleAuthResult($auth) utilities_auth2.php Sets legacy globals ($username, etc.) or redirects/dies on failure.
getUserCourseAccessDetails() utilities_authentication.php Returns security level (-1 to 3) for a user in a course.
get_current_username_or_ip() utilities_authentication.php Resolves the identity/username (using Auth2) or IP for logging.

9. Security Concerns

🚨 Whitelist F Fallback Risk

Issue: Unlisted PHP files can execute without login.

Fix: Change fallback to checkUserLoginAndSetUserInfo(false) or deny access entirely.

⚠️ SPC (Student Pin Code) Handling

Issue: Previously, the SPC parameter could bypass access checks.

Status: Fixed in router_api.php and utilities_authentication.php. Instructor "Student View" may need testing.

ℹ️ Role-Based Config

Status: COMPLETED (March 2026). Whitelists are now structured by role in page_whitelists.php and enforced by router_security.php.

10. Architecture Benefits

✅ DRY Page Logic

Pages no longer duplicate access checks. Removing 100+ lines of redundant if (!is_admin()) across the codebase.

ℹ️ Declarative Security

Security is now "declared" in the config rather than implemented in code, making audits significantly easier.
